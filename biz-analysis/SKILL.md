# Biz Analysis — 深度业务分析框架

## 什么时候用这个 skill

当用户说「帮我分析 XX 业务」「我想研究 XX 的问题」「XX 团队让我帮他们看看」时触发。

这个 skill 封装了一套经过验证的分析框架，产出物是**有数据支撑、有竞品对标、有可执行优先级**的分析报告，而不是泛泛的建议。

---

## 第一步：收集输入（必须做，不能跳过）

在开始分析之前，先向用户要以下文档。**文档越多，结论越有力。**

### 必须有（缺少则明确告知用户影响）

| 文档类型 | 具体内容 | 没有会怎样 |
|---------|---------|----------|
| **历史效果数据** | 核心指标的实际数据（ARPU、转化率、留存、收入峰值等） | 结论没有数据支撑，说服力弱 |
| **产品/业务背景** | 当前业务模式、核心 KPI、用户结构 | AI 只能给通用答案，无法贴合实际 |
| **问题或目标描述** | 现在遇到什么瓶颈，想解决什么 | 分析方向可能偏 |

### 加分项（有则更好）

| 文档类型 | 说明 |
|---------|------|
| 竞品信息 | 竞品的玩法、数据、机制（能大幅提升结论可信度） |
| 会议纪要 / 调研记录 | 一线用户/团队的真实反馈 |
| 历史复盘 / 失败案例 | 「差的案例」往往比成功案例更有价值 |
| 内部工具/流程文档 | 帮助识别 AI 盲区（AI 不知道你们内部系统长什么样） |

---

## 第二步：构建分析 Prompt

用以下结构组织给 AI 的输入：

```
【背景】
{产品是什么} + {核心变现模式} + {目标用户} + {当前核心 KPI 数值}

例：
天天练是 K12 学习产品，活动是最主要收入来源，付费率约 4%。
活动效果的核心变量是皮肤——用户喜不喜欢皮肤直接决定付费。

【目标】（按优先级排列）
目标1（最核心）：{主要问题}
目标2：{次要问题}
目标3：{附加问题}

【约束/已知信息】
- {已有的假设或限制条件}
- {团队的疑问或争议点}
```

**关键原则：**
- 背景要包含具体数字，不要用「还不错」「一般」这类模糊描述
- 目标必须分优先级，AI 会按权重分配分析深度
- 把你的疑问或争议点写进去，AI 会重点处理

---

## 第三步：标准分析结构（默认输出框架）

每次分析报告必须包含以下模块，缺少任何一项都要补上：

### 必须有的模块

```
1. 全景梳理
   - 历史时间线 / 事件全览（让 AI 先建立完整地图再分析）

2. 效果对比（好的 vs 差的，各举 3 个）
   - 不只说结果，要说「为什么好」「为什么差」的根因
   - 每个结论必须有数据支撑（内部数据或竞品数据）

3. 规律提炼（铁律 / 结论）
   - 从多个数据点中提炼出可复用的规则
   - 用数字说话，不用形容词

4. 竞品对标
   - 至少 3 个竞品的实际数据（不是「竞品也在做类似的事」这种废话）
   - 聚焦「我们能学什么」而不是「他们在做什么」

5. 用户分层
   - 不同用户群对分析对象的反应差异
   - 对应的差异化策略

6. 可执行优先级（必须按时间分层）
   - 🔴 本周可以做的
   - 🟡 2-4 周内
   - 🟢 1-3 个月
   - 每项必须写：预期效果 + 数据支撑 + 开发/执行成本估计
```

---

## 第四步：迭代循环（最重要的一步）

第一稿出来后，**不要直接用**。按以下顺序处理：

### 质疑清单（逐条问 AI）

1. **「这个结论是怎么从数据推导出来的？」**
   - 如果 AI 给了结论但没有推导过程，要求它重做并展示数据链条

2. **「这个建议有没有被竞品验证过？」**
   - 让 AI 找竞品案例支撑，如果找不到，这个建议要降优先级

3. **「最差的案例，你找到根因了吗？」**
   - 坏案例的根因分析比成功案例更有价值

4. **「这个结论在我们的实际情况下能不能用？」**
   - AI 不知道你的内部限制、团队结构、系统架构，需要你判断

### 盲区识别（你要主动做）

AI 不会告诉你它不知道什么。你要主动判断：

- **AI 不知道你的内部系统**（后台结构、配置流程、系统限制）→ 生成或导入操作手册
- **AI 不知道你们团队的真实执行能力**（什么1周能做，什么3个月都做不完）→ 你来评估
- **AI 不知道内部政治和决策链路**（谁的 KPI，谁能拍板）→ 你来判断优先级

---

## 第五步：输出核验

在把报告给业务方之前，检查：

- [ ] 每个结论后面有数据（内部数据 or 竞品数据）
- [ ] 有实际数字，不只有百分比描述
- [ ] 优先级清单是可以直接开工的（不是「应该做 AB 测试」而是「用 X 和 Y 做 AB 测试，时间窗口是 Z，主指标是 W」）
- [ ] 识别了分析的局限性（哪些结论是假设，哪些需要验证）
- [ ] 竞品数据有来源（不是凭空说的）

---

## 快速应用模板

对新业务快速启动，直接用这个 prompt 模板：

```
我需要你帮我深度分析 [业务名称] 的 [核心问题]。

背景：
- 产品是什么：[一句话描述]
- 核心变现模式：[如何赚钱]
- 目标用户：[谁在用]
- 当前核心 KPI：[具体数字]
- 当前卡点：[遇到什么问题]

我会提供以下材料：[列出你有的文档]

分析目标（按优先级）：
1. [最核心的问题]
2. [次要问题]
3. [附加问题]

要求：
- 每个结论必须有数据支撑（内部数据或竞品数据）
- 好的案例和差的案例各举3个，并分析根因
- 提炼可复用的规律（铁律）
- 给出竞品对标（至少3个竞品的实际数据）
- 优先级分层：本周 / 2-4周 / 1-3个月
```

---

## 这套方法适用的场景

| 场景 | 调整点 |
|------|--------|
| 商业化活动分析 | 重点放在效果数据 + 机制对标 |
| 用户增长分析 | 重点放在漏斗数据 + 用户分层 |
| 产品功能评估 | 重点放在使用数据 + 竞品对比 |
| 运营策略复盘 | 重点放在时间线全景 + 根因分析 |
| 新业务调研 | 重点放在竞品深度 + 行业规律 |
